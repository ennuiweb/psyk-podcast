# Podcasts Memory

## Snapshot

- Feed model note, 2026-04-19: the shared feed pipeline now supports both `storage.provider=drive` and `storage.provider=r2`. Feed generation uses `podcast-tools/storage_backends.py`, preserves GUID continuity from `episode_inventory.json`, and can read Cloudflare R2 either from a manifest or direct bucket listing.
- `podcast-tools/gdrive_podcast_feed.py` now merges storage-backed media inventory with repo-authored inputs such as `episode_metadata.json`, `reading_summaries.json`, `weekly_overview_summaries.json`, `quiz_links.json`, `auto_spec.json`, important-text docs, filters, and feed block rules before writing RSS.
- `podcast-tools/transcode_drive_media.py` remains Drive-only by design and exits cleanly for object-storage-backed feed reads; `scripts/migrate_drive_show_to_r2.py` is now the canonical exception for `personal`, where the importer can transcode configured source formats such as `.m4a` and `.wav` to MP3 before uploading stable R2 objects and backfill missing manifest checksums on resumed catalogs.
- For Freudd-backed subjects, `content_manifest.json` is the lecture-first merge layer built from reading key + quiz links + RSS + Spotify map + slides catalog; the portal reads that manifest instead of treating Drive as the UI source of truth.
- Migration program note, 2026-04-29: the repo now tracks the combined Hetzner NotebookLM queue + R2/object-storage migration as one implementation program in `docs/notebooklm-queue-r2-migration.md`, with an external tracker in Google Sheets. `bioneuro` is now the first live queue-owned R2 show, `personal` is now live on R2 under the legacy workflow, and `personlighedspsykologi-en` is now live on R2 under the legacy workflow as the final pre-queue subject.
- Queue status note, 2026-05-01: the repo now also keeps a separate current-state checkpoint in `docs/notebooklm-queue-current-state.md` so future sessions can distinguish shipped queue/R2 work from the longer-term migration backlog.
- Queue runtime note, 2026-05-03: the queue now also has a service-oriented `drain-show` entrypoint plus checked-in Hetzner `systemd` artifacts and an install/runbook in `docs/notebooklm-queue-operations.md`; the remaining gap is server installation/verification, not queue stage design.
- Queue core note, 2026-05-01: `notebooklm_queue/` now covers durable queue storage, adapter-based discovery for `bioneuro` and `personlighedspsykologi-en`, dry-run planning, real queue-owned generate/download execution with persisted run manifests, publish-bundle preparation, R2 object upload, queue-owned quiz-link refresh, repo metadata rebuild, allowlisted repo commit/push, and downstream workflow observation via `scripts/notebooklm_queue.py`. Successful jobs can now advance from `awaiting_publish` to `approved_for_publish` to `objects_uploaded` to `committing_repo_artifacts` to `repo_pushed` to `completed`, with deterministic object keys, upload verification, repo-side R2 media manifest refresh, quiz sync, RSS/inventory regeneration, Spotify sync, Freudd content-manifest rebuild, pushed generated repo artifacts, and validation of expected push-triggered downstream workflows; production cutover is still pending.
- Feed ownership note, 2026-05-03: `shows/*/config.github.json` now supports `publication.owner`, and `.github/workflows/generate-feed.yml` respects it. `legacy_workflow` remains the default, while queue-owned shows can now skip the legacy writer path entirely without changing the rest of the matrix.
- Personligheds hardening note, 2026-05-04: queue metadata rebuild for `personlighedspsykologi-en` now enforces manual summary completeness with a strict validator, runs `sync_regeneration_registry.py` as part of the publish contract, validates registry/inventory alignment after feed generation, and blocks on slide-brief coverage failures instead of treating them as warn-only queue output. The show still remains on `publication.owner = "legacy_workflow"` until its storage cutover is proven.
- Personligheds storage-cutover note, 2026-05-04: `personlighedspsykologi-en` is now live on `storage.provider = "r2"` while staying on `publication.owner = "legacy_workflow"`. The legacy workflow imports only the currently published Drive-backed inventory into `shows/personlighedspsykologi-en/media_manifest.r2.json`, preserves original `source_drive_file_id` values so regeneration validation can keep using Drive identities, and feed generation now prefers storage-level public URL settings for R2-backed shows even when the top-level config still carries a legacy Drive `public_link_template` for source-import compatibility.
- Queue pilot note, 2026-05-03: queue discovery and publication stages now accept an alternate `--show-config`, and the publish manifest pins that selected config path through upload, metadata rebuild, and repo push. This allows an R2 pilot for `bioneuro` to run against a non-live config without flipping the production `config.github.json` first.
- Queue pilot safety note, 2026-05-03: queue metadata rebuild, repo publication, and Freudd content-manifest rebuild now derive feed-side artifact paths from the selected show config instead of hardcoded live `shows/<show>/...` paths. `shows/bioneuro/config.r2-pilot.template.json` is now the checked-in safe template for a non-live `bioneuro` R2 pilot, but it still needs a real `storage.public_base_url` before it can be used end to end.
- Live cutover note, 2026-05-03: `bioneuro` is now the first live queue-owned R2-backed show in the repo. `shows/bioneuro/config.github.json` now resolves to `storage.provider=r2` and `publication.owner=queue`, the full `29`-episode back-catalog has been seeded into `shows/bioneuro/media_manifest.r2.json` with preserved `stable_guid` values from the legacy Drive inventory, and a live queue-owned publication completed successfully with downstream Freudd deploy success. The current public enclosure base is the temporary Cloudflare hostname `https://pub-fe942499398a478c8a8f432207051244.r2.dev`.
- Notebook creation note, 2026-04-30: `notebooklm-podcast-auto/generate_podcast.py` now tries to reclaim account capacity on `CREATE_NOTEBOOK` failures by deleting the oldest safe owned notebook on that account and retrying notebook creation once before falling back to profile rotation. Reclaim skips notebooks with pending artifacts or local request logs whose target output is still missing.
