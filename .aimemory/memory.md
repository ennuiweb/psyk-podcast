# Podcasts Memory

## Snapshot

- Feed model note, 2026-04-19: the shared feed pipeline now supports both `storage.provider=drive` and `storage.provider=r2`. Feed generation uses `podcast-tools/storage_backends.py`, preserves GUID continuity from `episode_inventory.json`, and can read Cloudflare R2 either from a manifest or direct bucket listing.
- Feed continuity note, 2026-05-04: `podcast-tools/gdrive_podcast_feed.py` now also falls back to the existing checked-in RSS identity map when a show migrates to R2 without a committed `episode_inventory.json`. That fallback was required to cut over `intro-vt` and `social-psychology` without GUID churn.
- `podcast-tools/gdrive_podcast_feed.py` now merges storage-backed media inventory with repo-authored inputs such as `episode_metadata.json`, `reading_summaries.json`, `weekly_overview_summaries.json`, `quiz_links.json`, `auto_spec.json`, important-text docs, filters, and feed block rules before writing RSS.
- `podcast-tools/transcode_drive_media.py` remains Drive-only by design and exits cleanly for object-storage-backed feed reads; `scripts/publish_local_show_to_r2.py` is now the canonical ingest path for `personal`, where the publisher can transcode configured source formats such as `.m4a` and `.wav` to MP3 before uploading stable R2 objects and backfill missing manifest checksums on resumed catalogs.
- For Freudd-backed subjects, `content_manifest.json` is the lecture-first merge layer built from reading key + quiz links + RSS + Spotify map + slides catalog; the portal reads that manifest instead of treating Drive as the UI source of truth.
- Migration program note, 2026-04-29: the repo now tracks the combined Hetzner NotebookLM queue + R2/object-storage migration as one implementation program in `docs/notebooklm-queue-r2-migration.md`, with an external tracker in Google Sheets. `bioneuro` is live on R2 and queue-owned, `personal` is live on R2 under the legacy workflow, and `personlighedspsykologi-en` is now also R2-backed and queue-owned.
- Queue status note, 2026-05-01: the repo now also keeps a separate current-state checkpoint in `docs/notebooklm-queue-current-state.md` so future sessions can distinguish shipped queue/R2 work from the longer-term migration backlog.
- Queue runtime note, 2026-05-04: the Hetzner queue wrapper now runs a quota-aware `serve-show` worker built on top of the single-cycle `drain-show` primitive. The service waits through `retry_scheduled` NotebookLM cooldown windows only when they are the sole remaining active backlog, resumes automatically when profile quota or transient NotebookLM create/generation failures become retryable again, fails closed on invalid retry schedules instead of hot-looping, and keeps the conservative timer-based runtime for fresh discovery.
- Queue core note, 2026-05-01: `notebooklm_queue/` now covers durable queue storage, adapter-based discovery for `bioneuro` and `personlighedspsykologi-en`, dry-run planning, real queue-owned generate/download execution with persisted run manifests, publish-bundle preparation, R2 object upload, queue-owned quiz-link refresh, repo metadata rebuild, allowlisted repo commit/push, and downstream workflow observation via `scripts/notebooklm_queue.py`. Successful jobs can now advance from `awaiting_publish` to `approved_for_publish` to `objects_uploaded` to `committing_repo_artifacts` to `repo_pushed` to `completed`, with deterministic object keys, upload verification, repo-side R2 media manifest refresh, quiz sync, RSS/inventory regeneration, Spotify sync, Freudd content-manifest rebuild, pushed generated repo artifacts, and validation of expected push-triggered downstream workflows; production cutover is still pending.
- Feed ownership note, 2026-05-03: `shows/*/config.github.json` now supports `publication.owner`, and `.github/workflows/generate-feed.yml` respects it. `legacy_workflow` remains the default, while queue-owned shows can now skip the legacy writer path entirely without changing the rest of the matrix.
- GitHub Actions runtime note, 2026-05-04: the repo workflows that still use shared JavaScript actions are now on `actions/checkout@v6` and `actions/setup-python@v6`, which removes the active Node 20 deprecation warning from the feed workflow path.
- Personligheds hardening note, 2026-05-04: queue metadata rebuild for `personlighedspsykologi-en` now enforces manual summary completeness with a strict validator, runs `sync_regeneration_registry.py` as part of the publish contract, validates registry/inventory alignment after feed generation, and blocks on slide-brief coverage failures instead of treating them as warn-only queue output.
- Personligheds learning-material ledger note, 2026-05-06: `shows/personlighedspsykologi-en/learning_material_regeneration_registry.json` is the checked-in record for which learner-facing podcasts, printouts, quizzes, and slide decks have been regenerated or published under the current prompt/Course Understanding pipeline. Queue metadata rebuild syncs it after content-manifest rebuild, repo publish allowlists it, podcasts are discovered from `episode_inventory.json`/`media_manifest.r2.json` even without local request logs, printouts carry generator setup `config_hash`/`config_fingerprint` plus separate Course Understanding provenance fingerprints, quiz entries preserve quiz-specific config hashes when local quiz artifacts exist while link-only published quizzes keep the source podcast config hash, and changed prompt/config identities retain previous observations in `revision_history`; use `scripts/sync_personlighedspsykologi_learning_material_registry.py` for manual checks.
- Personligheds ownership note, 2026-05-04: `personlighedspsykologi-en` now resolves to `storage.provider = "r2"` and `publication.owner = "queue"`. Its checked-in `media_manifest.r2.json` is the canonical published-audio inventory, while preserved `source_drive_file_id` fields remain historical compatibility metadata for regeneration validation.
- Personligheds runtime note, 2026-05-04: the Hetzner queue runtime is now also installed for `personlighedspsykologi-en` at `/etc/podcasts/notebooklm-queue/personlighedspsykologi-en.env` with `podcasts-notebooklm-queue@personlighedspsykologi-en.timer` enabled; the first manual drain completed as an expected no-op because discovery skips lecture keys already present in the live inventory.
- Active-audio surface note, 2026-05-04: `intro-vt` and `social-psychology` are also live on `storage.provider = "r2"` under `publication.owner = "legacy_workflow"`, and the active publication surface no longer depends on Drive source ingest. Remaining Drive-only tooling is limited to paused legacy feeds and historical migration helpers.
- Queue pilot note, 2026-05-03: queue discovery and publication stages now accept an alternate `--show-config`, and the publish manifest pins that selected config path through upload, metadata rebuild, and repo push. This allows an R2 pilot for `bioneuro` to run against a non-live config without flipping the production `config.github.json` first.

The current public enclosure base is the temporary Cloudflare hostname `https://pub-fe942499398a478c8a8f432207051244.r2.dev`.
- Notebook creation note, 2026-04-30: `notebooklm-podcast-auto/generate_podcast.py` now tries to reclaim account capacity on `CREATE_NOTEBOOK` failures by deleting the oldest safe owned notebook on that account and retrying notebook creation once before falling back to profile rotation. Reclaim skips notebooks with pending artifacts or local request logs whose target output is still missing.
