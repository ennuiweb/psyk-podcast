# Podcasts Memory

## Snapshot

- Feed model note, 2026-04-19: the shared feed pipeline now supports both `storage.provider=drive` and `storage.provider=r2`. Feed generation uses `podcast-tools/storage_backends.py`, preserves GUID continuity from `episode_inventory.json`, and can read Cloudflare R2 either from a manifest or direct bucket listing.
- `podcast-tools/gdrive_podcast_feed.py` now merges storage-backed media inventory with repo-authored inputs such as `episode_metadata.json`, `reading_summaries.json`, `weekly_overview_summaries.json`, `quiz_links.json`, `auto_spec.json`, important-text docs, filters, and feed block rules before writing RSS.
- `podcast-tools/transcode_drive_media.py` remains Drive-only by design and exits cleanly for object-storage-backed shows; the intended R2 path is to upload already-publishable audio at stable object keys.
- For Freudd-backed subjects, `content_manifest.json` is the lecture-first merge layer built from reading key + quiz links + RSS + Spotify map + slides catalog; the portal reads that manifest instead of treating Drive as the UI source of truth.
- Migration program note, 2026-04-29: the repo now tracks the combined Hetzner NotebookLM queue + R2/object-storage migration as one implementation program in `docs/notebooklm-queue-r2-migration.md`, with an external tracker in Google Sheets. Planned migration order is `personal`, then `bioneuro`, then `personlighedspsykologi-en`.
- Queue status note, 2026-05-01: the repo now also keeps a separate current-state checkpoint in `docs/notebooklm-queue-current-state.md` so future sessions can distinguish shipped queue/R2 work from the longer-term migration backlog.
- Queue core note, 2026-05-01: `notebooklm_queue/` now covers durable queue storage, adapter-based discovery for `bioneuro` and `personlighedspsykologi-en`, dry-run planning, real queue-owned generate/download execution with persisted run manifests, publish-bundle preparation, R2 object upload, queue-owned quiz-link refresh, repo metadata rebuild, and allowlisted repo commit/push via `scripts/notebooklm_queue.py`. Successful jobs can now advance from `awaiting_publish` to `approved_for_publish` to `objects_uploaded` to `committing_repo_artifacts` to `repo_pushed`, with deterministic object keys, upload verification, repo-side R2 media manifest refresh, quiz sync, RSS/inventory regeneration, Spotify sync, Freudd content-manifest rebuild, and pushed generated repo artifacts; downstream sync and production cutover are still pending.
- Notebook creation note, 2026-04-30: `notebooklm-podcast-auto/generate_podcast.py` now tries to reclaim account capacity on `CREATE_NOTEBOOK` failures by deleting the oldest owned notebook on that account and retrying notebook creation once before falling back to profile rotation.
